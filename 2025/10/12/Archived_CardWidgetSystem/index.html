Card Widget System

Card Widget System — Modular UI Framework for Game Prototyping

Card Widget System is a modular, data-driven user interface framework built in Unreal Engine 5 to streamline the creation and testing of card-based mechanics in strategy and simulation games. It enables designers to rapidly prototype card interactions, effects, and states without rewriting logic or rebuilding the interface from scratch.

Concept and Purpose

This project was developed as part of my Capstone, with the goal of bridging system design and tool engineering. It treats the card as both a gameplay entity and a dynamic UI component, reflecting real-time data while maintaining synchronization with the backend system.

The system is scalable, reusable, and adaptable for any project requiring state-driven UI elements — from tactical card games to crafting interfaces or simulation dashboards.

Role, Responsibilities, Tools

Role: System and UI Designer
Responsibilities: Architecture design, UI state synchronization, data table management, effect and slot system development, documentation for reuse.
Tools: Unreal Engine 5, UMG, Blueprint, DataTables, Visual Studio, GitHub, Miro, and Excel for data-driven testing.

System Architecture Overview

Core Structure

Card Object Layer: Stores all gameplay data, including card attributes, tags, cooldowns, and effects.

UI Widget Layer: Handles rendering, user input, animation, and dynamic binding with card data.

Controller Layer: Manages drag-and-drop interactions, effect resolution, and card targeting logic.

DataTable Integration: Provides editable balance sheets for non-programmers to tune parameters quickly.

Design Principle:

Each layer functions independently but communicates through data binding and event dispatchers, enabling real-time updates across systems with minimal coupling.

Design Highlights

  1. Modularity
    Every UI element — from icons to card borders — is modular and replaceable. Designers can plug new components into the system using standardized interfaces.

  2. Scalability
    The system supports hundreds of card instances simultaneously without performance loss through efficient data caching and event filtering.

  3. Reusability
    All Blueprint logic is encapsulated, allowing new projects to import the system with minimal setup. Designers can define card categories, states, and attributes directly from spreadsheets.

  4. Extensibility
    Supports effect chaining, combo triggers, and slot-based dependencies for complex system behaviors.

Example Workflow

Import card metadata via DataTable (CSV or XLSX).

Assign each card a Widget Blueprint derived from the base CardWidget class.

Connect UI bindings to gameplay data (e.g., mana cost, description, cooldown).

Use the system’s global dispatcher to sync animation and logic events in real time.

Test in editor with debug overlays showing card state transitions.

Gameplay Integration

The Card Widget System can function as:

A combat interface for deck-building or tactical RPGs.

A crafting interface that visualizes recipes as interactive cards.

A system design testing tool for simulating event-driven logic.

In its Capstone version, it served as a prototype environment for simulating combat card interactions, allowing quick iteration of damage, buff, and cost parameters.

Design Goals

Speed: Rapid prototyping without reimplementing UI or data structures.

Consistency: All cards share the same logic core and visual feedback standards.

Adaptability: Easily connectable with different gameplay genres.

Documentation: Fully annotated for reuse by other designers and students.

Research Summary

This project applies the principles of systemic modularity and feedback loop clarity from the MDA framework (Hunicke et al., 2004). Fullerton’s (2018) iterative design model guided the creation of repeatable test cycles to verify usability and stability. Concepts from Salen and Zimmerman (2004) supported the structuring of consistent visual feedback for meaningful interaction, while Deterding et al. (2011) inspired the reward pacing and affordance structure of the interface.
